# v0.1 Scaffold Technical Design

Status: Initial scaffold with provider setup UI
Last updated: 2026-06-19

## Purpose

This scaffold turns the PRD's core scenario into a runnable local app path
before integrating real external runtimes.

The goal is not to complete the production adapter stack. The goal is to prove
the product event loop:

```text
natural-language task
  -> plan
  -> task-scoped permission
  -> progress events
  -> artifact preview
  -> task history
```

## Current Implementation

Implemented:

- Tauri 2 app shell.
- Vite + React + TypeScript frontend.
- Dark console-inspired Wutai UI.
- Core research task entry.
- Generated plan.
- Task-scoped permission prompt.
- Permission approval and denial paths.
- Mock research adapter that emits Wutai task events.
- Mock `report.md`, `sources.json`, and `audit.json` artifacts.
- Adapter interface and runtime adapter factory.
- Local task history through a `TaskStore` interface.
- SQLite-backed `TaskStore` in the Tauri runtime using `tauri-plugin-sql`.
- Browser fallback `TaskStore` backed by `localStorage` for web preview and
  e2e tests.
- Tauri artifact writer that writes generated task artifacts to the app-data
  directory.
- Optional GPT Researcher sidecar path behind a Tauri command.
- GPT Researcher setup preflight for Python, sidecar script, package, and API
  keys.
- Non-technical Research setup form for model and web-search access keys.
- System keychain-backed credential storage through `keyring-rs`.
- Task-scoped GPT Researcher sidecar cancellation.
- GPT Researcher stderr capture as expert-only `ToolLogAdded` task events.
- Rust command-logic tests for credential storage behavior, setup-key
  precedence, and task-scoped sidecar cancellation.

Not implemented yet:

- GPT Researcher is not the default runtime yet; it is opt-in through
  `VITE_WUTAI_RESEARCH_ADAPTER=gpt-researcher`.
- Streaming progress from GPT Researcher internals.
- Browser-use adapter.
- Codex app-server adapter.
- Computer-use adapter.
- Voice.

## Adapter Boundary

The UI does not know whether events come from a mock adapter, GPT Researcher,
browser-use, Codex app-server, or a future computer-use runtime.

Every adapter should emit Wutai events:

- `TaskStarted`
- `TaskStepUpdated`
- `PermissionRequested`
- `PermissionResolved`
- `ArtifactCreated`
- `SourceCaptured`
- `ToolLogAdded`
- `TaskCompleted`
- `TaskFailed`

The default UI only renders `user` visibility events.

The current adapter selection is intentionally hidden from normal users. During
development, set:

```bash
VITE_WUTAI_RESEARCH_ADAPTER=gpt-researcher
```

to route approved research tasks through the Tauri `run_gpt_researcher`
command. If this variable is absent, Wutai uses the mock adapter so the core UI
and e2e tests remain deterministic and offline.

## GPT Researcher Sidecar

The sidecar uses the official `gpt-researcher` Python package rather than
rebuilding research orchestration. The Tauri command launches:

```text
scripts/gpt_researcher_adapter.py
```

The script runs `GPTResearcher`, collects `report`, source URLs, research
sources, and costs, then returns a JSON payload shaped for Wutai. Runtime logs
are redirected to stderr so stdout remains parseable JSON.

Tauri tracks each running sidecar process by `task_id`. The frontend Stop
control aborts the adapter run and calls `cancel_gpt_researcher`, which attempts
to terminate the matching Python process. When the sidecar completes
successfully, captured stderr lines are attached to the task as expert-only
`ToolLogAdded` events and included in `audit.json` under `sidecarLogs`.

Developer setup:

```bash
python3.11 -m venv .venv
. .venv/bin/activate
python -m pip install -r requirements-gpt-researcher.txt
export WUTAI_GPT_RESEARCHER_PYTHON="$PWD/.venv/bin/python"
VITE_WUTAI_RESEARCH_ADAPTER=gpt-researcher npm run tauri dev
```

When the app opens, use Research setup to save the model access key and web
search key. Wutai stores these secrets in the system keychain through
`keyring-rs` and injects them into the GPT Researcher sidecar process. The
developer environment variables `OPENAI_API_KEY` and `TAVILY_API_KEY` remain
fallbacks for scripted runs.

The adapter script path can be overridden with
`WUTAI_GPT_RESEARCHER_ADAPTER_SCRIPT`.

## Setup Preflight

When the GPT Researcher adapter is selected, Wutai calls the Tauri
`check_gpt_researcher` command during startup. The command returns:

- `ready`
- `summary`
- per-check setup status
- plain-language fixes
- detected Python path, script path, and GPT Researcher package version when
  available

The preflight currently checks:

- local sidecar script availability
- Python launchability
- `gpt-researcher` package availability for that Python
- model access key from Wutai setup or `OPENAI_API_KEY`
- web search key from Wutai setup or `TAVILY_API_KEY`

If any required check fails, Wutai blocks creation of new real research tasks
and shows the setup summary in the task console. The mock adapter returns a
ready preflight with no visible setup panel, so the default offline workflow
stays unchanged.

The credential and cancellation logic behind these Tauri commands has Rust
tests. Credential tests use an in-memory implementation of the same storage
interface and do not read or write the user's system keychain. The process test
starts a local sleeping child process, cancels it by task ID, and verifies
registry cleanup. These tests do not yet exercise a command through a real
Tauri webview IPC boundary.

## Storage Boundary

The scaffold uses SQLite when running in Tauri and `localStorage` when running
as a plain web app.

The UI depends on the `TaskStore` interface rather than the storage backend.
This keeps the browser preview testable while allowing the desktop shell to use
real local persistence.

## Artifact Boundary

The mock adapter still generates artifact content in memory. When running in
Tauri, `artifactWriter` calls the `write_task_artifacts` command and writes
files to:

```text
<app-data-dir>/artifacts/<task_id>/
```

When running as a plain web app, artifacts remain available in preview and
download mode only.
