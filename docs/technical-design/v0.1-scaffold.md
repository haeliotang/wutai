# v0.1 Scaffold Technical Design

Status: Initial scaffold  
Last updated: 2026-06-19

## Purpose

This scaffold turns the PRD's core scenario into a runnable local app path
before integrating real external runtimes.

The goal is not to complete the production adapter stack. The goal is to prove
the product event loop:

```text
natural-language task
  -> plan
  -> task-scoped permission
  -> progress events
  -> artifact preview
  -> task history
```

## Current Implementation

Implemented:

- Tauri 2 app shell.
- Vite + React + TypeScript frontend.
- Dark console-inspired Wutai UI.
- Core research task entry.
- Generated plan.
- Task-scoped permission prompt.
- Permission approval and denial paths.
- Mock research adapter that emits Wutai task events.
- Mock `report.md` and `sources.json` artifacts.
- Local task history through a `TaskStore` interface backed by `localStorage`.

Not implemented yet:

- SQLite-backed storage.
- GPT Researcher sidecar/adapter.
- Real web search.
- Real file artifact writes.
- Browser-use adapter.
- Codex app-server adapter.
- Computer-use adapter.
- Voice.

## Adapter Boundary

The UI does not know whether events come from a mock adapter, GPT Researcher,
browser-use, Codex app-server, or a future computer-use runtime.

Every adapter should emit Wutai events:

- `TaskStarted`
- `TaskStepUpdated`
- `PermissionRequested`
- `PermissionResolved`
- `ArtifactCreated`
- `SourceCaptured`
- `ToolLogAdded`
- `TaskCompleted`
- `TaskFailed`

The default UI only renders `user` visibility events.

## Storage Boundary

The scaffold uses `localStorage` only to keep the first app path runnable in a
browser and Tauri webview.

The next persistence milestone should replace `localTaskStore` with a
SQLite-backed implementation. The UI should continue to depend on the
`TaskStore` interface rather than the storage backend.

## Next Engineering Milestone

Replace mock persistence with SQLite and add a real artifact writer:

1. Add a SQLite-backed `TaskStore`.
2. Store `tasks`, `task_events`, `permission_requests`, `sources`, and
   `artifacts`.
3. Write report artifacts to an approved local output directory.
4. Keep the current mock adapter working against the new store.

After that, integrate GPT Researcher behind the same adapter boundary.
