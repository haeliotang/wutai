# v0.1 Scaffold Technical Design

Status: Initial supervised research scaffold with Provider Profiles and v0.2 CLI wrapper
Last updated: 2026-06-27

## Purpose

This scaffold turns the PRD's core supervised research scenario into a runnable
local app path before integrating broader external agent runtimes.

The goal is not to complete the production adapter stack. The goal is to prove
the local trust-layer event loop:

```text
natural-language task
  -> plan
  -> task-scoped permission
  -> progress events
  -> work packet
  -> evidence review
  -> task history
```

  e2e tests.
- Tauri artifact writer that writes generated task artifacts to the app-data
  directory.
- Optional GPT Researcher sidecar path behind a Tauri command.
- GPT Researcher setup preflight for Python, sidecar script, package, the
  active Provider Profile, required API keys, and Ollama reachability.
- Provider Profiles with separate model, search, and embedding choices and
  advanced endpoint controls.
- System keychain-backed credential storage through `keyring-rs`.

## Adapter Boundary

The UI does not need to know whether events come from a mock adapter,
GPT Researcher, a CLI wrapper, MCP proxy, browser-use, Codex app-server,
Claude Code trace importer, or a future computer-use runtime.

Every adapter should emit Wutai events:

- `TaskStarted`
- `TaskStepUpdated`
- `PermissionRequested`
- `PermissionResolved`
- `HumanConfirmationNeeded`
- `ArtifactCreated`
- `SourceCaptured`
- `ToolCallCaptured`
- `RuntimeEventCaptured`
- `CredentialGrantRecorded`
- `ToolLogAdded`
- `TaskCompleted`
- `TaskFailed`

The default UI only renders `user` visibility events.

Adapters are expected to preserve Wutai's local ledger and permission boundary.
If a runtime cannot be constrained by Wutai, it must be marked experimental and
kept out of the default path.

The current adapter selection is intentionally hidden from normal users. During
development, set:

```bash
VITE_WUTAI_RESEARCH_ADAPTER=gpt-researcher
```

to route approved research tasks through the Tauri `run_gpt_researcher`
command. If this variable is absent, Wutai uses the mock adapter so the core UI
and e2e tests remain deterministic and offline.

The local-script trace importer is a separate v0.2 wedge. It accepts a declared
trace for an already-run command and turns it into Wutai events, audit data, and
a work packet. It does not run the command, enforce shell permissions, watch the
filesystem, or prove that the trace is complete.

The coding-agent trace importer is the first external-agent packet path. It
accepts a declared `wutai.coding_agent_trace` JSON file with `agentName`,
`agentRuntime`, `title`, `userRequest`, repository, timestamps, status, summary,
tool calls, file changes, produced artifacts, credential purposes, and
limitations. Wutai converts it into a `coding_agent` work packet with
`report.md`, `trace.json`, `audit.json`, and `manifest.json`. The importer
records tool calls, file-change summaries, runtime summary, credential purposes,
and artifact hashes for review. It does not execute or supervise the coding
agent, verify trace completeness, enforce tool permissions, or capture file
diffs unless they are declared in the trace.

The MCP tool-call trace importer accepts a declared
`wutai.mcp_tool_call_trace` JSON file with client/server labels, timestamps,
status, summary, MCP tool calls, resources, credential purposes, and
limitations. Wutai converts it into an `mcp_tool_call` work packet with
`report.md`, `trace.json`, `audit.json`, and `manifest.json`. The importer
records declared server/tool names, request summaries, bounded argument previews,
result summaries, resources, runtime summary, credential purposes, and artifact
hashes. It does not proxy the MCP connection, execute or replay tools, approve
tool calls, mediate credentials, or prove trace completeness. The parser rejects
missing required tool names, invalid status or timestamp order, negative
latency, and overlarge tool-call batches before creating a task.

The local file-ingestion path accepts files explicitly selected by the user and
converts them into a `local_file` work packet with `report.md`, `files.json`,
`audit.json`, and `manifest.json`. It records file labels, MIME type, byte size,
SHA-256, bounded text previews, and file-read audit entries. It enforces bounded
file count and per-file size limits. A later re-check compares newly selected
files against the recorded SHA-256 and byte size, then writes `file-check.json`.
It does not crawl directories, watch future changes, retain full file contents,
or grant file access to a downstream agent.

The developer CLI wrapper is the first actual local command execution wedge:

```bash
npm run wutai:run -- -- <command> [args...]
```

It uses `child_process.spawn` with `shell: false`, records argv, working
directory, structured policy preflight, exit code, bounded stdout/stderr
summaries, git-status delta, and artifact hashes, then writes a work packet
under `artifacts/cli/<session_id>/`. With `--signing-key <pem>`, it also writes
an optional `attestation.json` that signs the final `manifest.json` bytes with
ECDSA P-256 SHA-256 and embeds the verification public key.

The policy preflight is a structured catalog with rule category, severity,
default action, overrideability, review scope, and decision rationale. It denies
matched high-risk patterns before execution unless `--allow-high-risk` is
provided, and records an optional `--override-reason`. Profile behavior is read
from `config/wutai-cli-policy-profiles.json` by default, with
`--policy-config <path>` available for custom profile sets. `--policy-profile
standard` preserves warning rules as warnings; `--policy-profile strict`
escalates warning rules to deny. Profiles can also define rule-level overrides
for individual rule ids, including effective action, severity, overrideability,
review scope, message, and reason. These overrides are recorded on matched rules
in `policy.json` and can strengthen or weaken a rule. `--dry-run` writes a
review packet without spawning the command; execution permission remains pending
and no tool call is recorded. Covered high-risk patterns include shell
interpreter command strings,
environment dumps, recursive or forced remove, privilege escalation commands,
destructive git operations, and recursive permission or ownership changes.
Medium-risk dependency installs, simple local network listeners, and Node
inspector listeners are recorded as warnings under the standard profile.

This is an execution ledger, not a sandbox. The wrapped process still inherits
the invoking shell's filesystem, environment, network, and credential access.
The wrapper does not enforce a complete destructive-command policy or mediate
secrets.

The desktop/web packet import path uses browser file inputs. The directory path
uses `webkitdirectory`; the fallback path accepts selected `manifest.json` and
sibling packet artifacts. Wutai validates that the manifest is a `wutaiRunCli`
local-script packet, converts the files into a `WutaiTask`, recomputes selected
artifact SHA-256 values against the manifest, persists an `integrity.json`
review artifact, persists a `provenance.json` artifact for manifest hash,
producer fields, required artifact presence, schema-kind consistency, and
optional attestation signature verification, and renders a CLI Packet Review
panel for policy, trace, ledger, filtered audit details, integrity, and
provenance. The UI can load a local `wutai.trusted_producer_policy` JSON file;
only keys in that local policy can turn a verified attestation into a trusted
producer result. If an attestation verifies but the key is unknown, the UI can
enroll the public key hash into local policy for the observed producer adapter
and packet type, then re-run packet provenance locally. Packet-supplied policy
files are not treated as trust roots. For dry-run packets with a pending
local-script execution permission, the panel can record approve or deny into
`review.json`. This is review-only; the UI does not execute or replay the
command, unsigned provenance is a warning, and local key enrollment is not
certificate-chain validation or remote identity proof.

## GPT Researcher Sidecar

The sidecar uses the official `gpt-researcher` Python package rather than
rebuilding research orchestration. The Tauri command launches:

```text
scripts/gpt_researcher_adapter.py
```

The script runs `GPTResearcher`, collects `report`, source URLs, research
sources, and costs, runs Evidence Gate, then returns a JSON payload shaped for
Wutai. Runtime logs are redirected to stderr so stdout remains parseable JSON.

Tauri tracks each running sidecar process by `task_id`. The frontend Stop
control aborts the adapter run and calls `cancel_gpt_researcher`, which attempts
to terminate the matching Python process. While the process runs, Tauri reads
stderr line by line and sends structured stage and log messages through its
official IPC Channel. The frontend turns stages into user-visible
`TaskStepUpdated` events and runtime lines into expert-only `ToolLogAdded`
events, with writes throttled to avoid one database transaction per line.

Task history keeps the first 200 expert log events. Before a line reaches the
Channel, error response, or audit payload, Tauri replaces exact provider-key
values with `[REDACTED]`. Every captured, redacted log line is still returned
by the sidecar command, bounded to 500 characters per line, and written to
`audit.json` under `sidecarLogs`. The final command payload also contains
sidecar stages so the frontend can reconcile any Channel messages that arrive
late without duplicating events.

Developer setup:

```bash
python3.13 -m venv .venv
. .venv/bin/activate
python -m pip install -r requirements-gpt-researcher.txt
VITE_WUTAI_RESEARCH_ADAPTER=gpt-researcher npm run tauri dev
```

The supported sidecar range is Python 3.11 through 3.13, with 3.13
recommended. Wutai automatically prefers the project `.venv`; use
`WUTAI_GPT_RESEARCHER_PYTHON` only as an explicit interpreter override.
The ignored `installed_gpt_researcher_sidecar_smoke` Rust test verifies the
optional environment's interpreter range, pinned package version, and core
import without loading credentials or starting research.

When the app opens, use Research setup to select or create a Provider Profile.
Each profile independently selects its model, search, and embedding providers.
The default profile maps DeepSeek to the GPT Researcher model layer, Tavily to
retrieval, and Ollama to local document embeddings. OpenAI,
OpenAI-compatible endpoints, DuckDuckGo, and fully local Ollama configurations
are also supported.

Non-secret profile metadata is persisted as
`research-provider-profiles.json` under the app-data directory. Wutai stores
keys in the system keychain through `keyring-rs`, scoped by profile, provider,
and purpose, then injects only the active profile into the sidecar process. The
developer environment variables `DEEPSEEK_API_KEY`, `OPENAI_API_KEY`, and
`TAVILY_API_KEY` remain fallbacks for scripted runs. Provider metadata, never
secret values, is attached to each successful run's audit payload.

The adapter script path can be overridden with
`WUTAI_GPT_RESEARCHER_ADAPTER_SCRIPT`.

## Evidence Gate v0.1

Evidence Gate runs after GPT Researcher drafts the report. It reuses the active
Provider Profile model to extract at most 30 decision-relevant claims into a
strict JSON shape. Model output is treated as untrusted input: Wutai drops
unknown fields and URLs, derives claim risk from fixed categories, classifies
source provenance with deterministic hostname rules, and computes the final
gate status itself.

High-risk claims include licenses, pricing, release dates, adoption metrics,
and product identity. These require both model-assessed support and a captured
primary or repository source to pass. Locked regressions currently reject the
known incorrect claims that OpenClaw is Apache 2.0 and that Multica uses
unmodified Apache 2.0. They are regression seeds, not a universal fact
database.

The gate returns `pass`, `warning`, or `fail`. A non-pass result does not discard
the report; the task becomes `completed_with_warnings`, the UI displays the
Evidence Gate summary, and `claims.json` plus `verification.json` preserve the
review surface. A pass means the extracted claims met the configured evidence
rules. It is not a guarantee that every statement in the report is true.

This is the first implementation of Wutai's work-packet model. It turns an
agent result into reviewable artifacts plus machine-readable sidecars, rather
than treating the final report as self-certifying.

## Setup Preflight

When the GPT Researcher adapter is selected, Wutai calls the Tauri
`check_gpt_researcher` command during startup. The command returns:

- `ready`
- `summary`
- per-check setup status
- plain-language fixes
- detected Python path, script path, and GPT Researcher package version when
  available

The preflight currently checks:

- local sidecar script availability
- Python launchability
- `gpt-researcher` package availability for that Python
- active Provider Profile validity
- required model access from the profile keychain slot or matching environment
  variable
- required search access from the profile keychain slot or matching environment
  variable
- required embedding access, including model-key reuse for OpenAI profiles
- Ollama endpoint reachability when the active profile uses Ollama

If any required check fails, Wutai blocks creation of new real research tasks
and shows the setup summary in the task console. The mock adapter returns a
ready preflight with no visible setup panel, so the default offline workflow
stays unchanged.

The credential, preflight, and cancellation commands have Rust tests through
Tauri's official mock runtime and WebView invoke handler. Credential tests use
an in-memory implementation of the same storage interface and do not read or
write the user's system keychain. The process test starts a local sleeping
child process, cancels it by task ID through IPC, and verifies termination and
registry cleanup. The suite does not call external model or search APIs and
does not run a real GPT Researcher task.

The streaming reader test uses an in-memory Tauri Channel. It verifies ordered
stage and log delivery, malformed stage fallback, provider-key redaction, and
preservation of captured audit lines without launching Python or external
services.

The Provider Profiles Playwright contract test supplies a mocked Tauri command
boundary, verifies the default and Advanced controls, switches model provider,
and checks the configured 860-pixel desktop minimum width for horizontal
overflow.

`tests/python/test_evidence_gate.py` runs without network or model access. It
locks source classification, fenced JSON parsing, primary-evidence warnings,
and OpenClaw/Multica license regressions.

`tests/node/wutai_run.test.mjs` runs the CLI wrapper against passing, failing,
policy-denied, and high-risk override local commands. It verifies that each path
writes a work packet, preserves failed child-process exit codes, and records the
policy decision.

## Storage Boundary

The scaffold uses SQLite when running in Tauri and `localStorage` when running
as a plain web app.

The UI depends on the `TaskStore` interface rather than the storage backend.
This keeps the browser preview testable while allowing the desktop shell to use
real local persistence.

## Artifact Boundary

The mock adapter still generates artifact content in memory. When running in
Tauri, `artifactWriter` calls the `write_task_artifacts` command and writes
files to:

```text
<app-data-dir>/artifacts/<task_id>/
```

When running as a plain web app, artifacts remain available in preview and
download mode only.

## Next Engineering Milestone

Harden the work-packet boundary before adding broader agent supervision:

1. Add an official-source-first research pass so Evidence Gate can improve
   weak retrieval instead of only flagging it after drafting.
2. Add fine-grained semantic progress only if GPT Researcher exposes a stable
   upstream event API; do not parse unstable human log wording.
3. Add more rule override regression and config validation, then harden MCP
   trace and local file-ingestion schema validation.
4. Keep the mock adapter available for offline UI/e2e validation.
