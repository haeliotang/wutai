# v0.1 Scaffold Technical Design

Status: Initial scaffold with GPT Researcher adapter boundary
Last updated: 2026-06-19

## Purpose

This scaffold turns the PRD's core scenario into a runnable local app path
before integrating real external runtimes.

The goal is not to complete the production adapter stack. The goal is to prove
the product event loop:

```text
natural-language task
  -> plan
  -> task-scoped permission
  -> progress events
  -> artifact preview
  -> task history
```

## Current Implementation

Implemented:

- Tauri 2 app shell.
- Vite + React + TypeScript frontend.
- Dark console-inspired Wutai UI.
- Core research task entry.
- Generated plan.
- Task-scoped permission prompt.
- Permission approval and denial paths.
- Mock research adapter that emits Wutai task events.
- Mock `report.md`, `sources.json`, and `audit.json` artifacts.
- Adapter interface and runtime adapter factory.
- Local task history through a `TaskStore` interface.
- SQLite-backed `TaskStore` in the Tauri runtime using `tauri-plugin-sql`.
- Browser fallback `TaskStore` backed by `localStorage` for web preview and
  e2e tests.
- Tauri artifact writer that writes generated task artifacts to the app-data
  directory.
- Optional GPT Researcher sidecar path behind a Tauri command.

Not implemented yet:

- GPT Researcher is not the default runtime yet; it is opt-in through
  `VITE_WUTAI_RESEARCH_ADAPTER=gpt-researcher`.
- Streaming progress from GPT Researcher internals.
- Cancellation of the running Python sidecar process after launch.
- Browser-use adapter.
- Codex app-server adapter.
- Computer-use adapter.
- Voice.

## Adapter Boundary

The UI does not know whether events come from a mock adapter, GPT Researcher,
browser-use, Codex app-server, or a future computer-use runtime.

Every adapter should emit Wutai events:

- `TaskStarted`
- `TaskStepUpdated`
- `PermissionRequested`
- `PermissionResolved`
- `ArtifactCreated`
- `SourceCaptured`
- `ToolLogAdded`
- `TaskCompleted`
- `TaskFailed`

The default UI only renders `user` visibility events.

The current adapter selection is intentionally hidden from normal users. During
development, set:

```bash
VITE_WUTAI_RESEARCH_ADAPTER=gpt-researcher
```

to route approved research tasks through the Tauri `run_gpt_researcher`
command. If this variable is absent, Wutai uses the mock adapter so the core UI
and e2e tests remain deterministic and offline.

## GPT Researcher Sidecar

The sidecar uses the official `gpt-researcher` Python package rather than
rebuilding research orchestration. The Tauri command launches:

```text
scripts/gpt_researcher_adapter.py
```

The script runs `GPTResearcher`, collects `report`, source URLs, research
sources, and costs, then returns a JSON payload shaped for Wutai. Runtime logs
are redirected to stderr so stdout remains parseable JSON.

Developer setup:

```bash
python3.11 -m venv .venv
. .venv/bin/activate
python -m pip install -r requirements-gpt-researcher.txt
export OPENAI_API_KEY=...
export TAVILY_API_KEY=...
export WUTAI_GPT_RESEARCHER_PYTHON="$PWD/.venv/bin/python"
VITE_WUTAI_RESEARCH_ADAPTER=gpt-researcher npm run tauri dev
```

The adapter script path can be overridden with
`WUTAI_GPT_RESEARCHER_ADAPTER_SCRIPT`.

## Storage Boundary

The scaffold uses SQLite when running in Tauri and `localStorage` when running
as a plain web app.

The UI depends on the `TaskStore` interface rather than the storage backend.
This keeps the browser preview testable while allowing the desktop shell to use
real local persistence.

## Artifact Boundary

The mock adapter still generates artifact content in memory. When running in
Tauri, `artifactWriter` calls the `write_task_artifacts` command and writes
files to:

```text
<app-data-dir>/artifacts/<task_id>/
```

When running as a plain web app, artifacts remain available in preview and
download mode only.
