# v0.1 Scaffold Technical Design

Status: Initial scaffold with Provider Profiles
Last updated: 2026-06-20

## Purpose

This scaffold turns the PRD's core scenario into a runnable local app path
before integrating real external runtimes.

The goal is not to complete the production adapter stack. The goal is to prove
the product event loop:

```text
natural-language task
  -> plan
  -> task-scoped permission
  -> progress events
  -> artifact preview
  -> task history
```

## Current Implementation

Implemented:

- Tauri 2 app shell.
- Vite + React + TypeScript frontend.
- Dark console-inspired Wutai UI.
- Core research task entry.
- Generated plan.
- Task-scoped permission prompt.
- Permission approval and denial paths.
- Mock research adapter that emits Wutai task events.
- Mock and real `report.md`, `sources.json`, `claims.json`,
  `verification.json`, and `audit.json` artifacts.
- Adapter interface and runtime adapter factory.
- Local task history through a `TaskStore` interface.
- SQLite-backed `TaskStore` in the Tauri runtime using `tauri-plugin-sql`.
- Browser fallback `TaskStore` backed by `localStorage` for web preview and
  e2e tests.
- Tauri artifact writer that writes generated task artifacts to the app-data
  directory.
- Optional GPT Researcher sidecar path behind a Tauri command.
- GPT Researcher setup preflight for Python, sidecar script, package, the
  active Provider Profile, required API keys, and Ollama reachability.
- Provider Profiles with separate model, search, and embedding choices and
  advanced endpoint controls.
- System keychain-backed credential storage through `keyring-rs`.
- Task-scoped GPT Researcher sidecar cancellation.
- GPT Researcher stderr streaming through a Tauri IPC Channel as expert-only
  `ToolLogAdded` task events.
- Structured sidecar stages translated into plain-language `TaskStepUpdated`
  events while research is running.
- Evidence Gate v0.1 claim extraction, deterministic source tiers, high-risk
  evidence checks, and locked license-fact regressions.
- Visible Evidence Gate result with `completed_with_warnings` when a generated
  report is usable but not ready to trust.
- Rust command and Tauri IPC tests for credential storage behavior, setup-key
  precedence, preflight, and task-scoped sidecar cancellation.

Not implemented yet:

- GPT Researcher is not the default runtime yet; it is opt-in through
  `VITE_WUTAI_RESEARCH_ADAPTER=gpt-researcher`.
- Fine-grained semantic progress from GPT Researcher internals beyond the
  current sidecar stages and runtime logs.
- Browser-use adapter.
- Codex app-server adapter.
- Computer-use adapter.
- Voice.

## Adapter Boundary

The UI does not know whether events come from a mock adapter, GPT Researcher,
browser-use, Codex app-server, or a future computer-use runtime.

Every adapter should emit Wutai events:

- `TaskStarted`
- `TaskStepUpdated`
- `PermissionRequested`
- `PermissionResolved`
- `ArtifactCreated`
- `SourceCaptured`
- `ToolLogAdded`
- `TaskCompleted`
- `TaskFailed`

The default UI only renders `user` visibility events.

The current adapter selection is intentionally hidden from normal users. During
development, set:

```bash
VITE_WUTAI_RESEARCH_ADAPTER=gpt-researcher
```

to route approved research tasks through the Tauri `run_gpt_researcher`
command. If this variable is absent, Wutai uses the mock adapter so the core UI
and e2e tests remain deterministic and offline.

## GPT Researcher Sidecar

The sidecar uses the official `gpt-researcher` Python package rather than
rebuilding research orchestration. The Tauri command launches:

```text
scripts/gpt_researcher_adapter.py
```

The script runs `GPTResearcher`, collects `report`, source URLs, research
sources, and costs, runs Evidence Gate, then returns a JSON payload shaped for
Wutai. Runtime logs are redirected to stderr so stdout remains parseable JSON.

Tauri tracks each running sidecar process by `task_id`. The frontend Stop
control aborts the adapter run and calls `cancel_gpt_researcher`, which attempts
to terminate the matching Python process. While the process runs, Tauri reads
stderr line by line and sends structured stage and log messages through its
official IPC Channel. The frontend turns stages into user-visible
`TaskStepUpdated` events and runtime lines into expert-only `ToolLogAdded`
events, with writes throttled to avoid one database transaction per line.

Task history keeps the first 200 expert log events. Before a line reaches the
Channel, error response, or audit payload, Tauri replaces exact provider-key
values with `[REDACTED]`. Every captured, redacted log line is still returned
by the sidecar command, bounded to 500 characters per line, and written to
`audit.json` under `sidecarLogs`. The final command payload also contains
sidecar stages so the frontend can reconcile any Channel messages that arrive
late without duplicating events.

Developer setup:

```bash
python3.13 -m venv .venv
. .venv/bin/activate
python -m pip install -r requirements-gpt-researcher.txt
VITE_WUTAI_RESEARCH_ADAPTER=gpt-researcher npm run tauri dev
```

The supported sidecar range is Python 3.11 through 3.13, with 3.13
recommended. Wutai automatically prefers the project `.venv`; use
`WUTAI_GPT_RESEARCHER_PYTHON` only as an explicit interpreter override.
The ignored `installed_gpt_researcher_sidecar_smoke` Rust test verifies the
optional environment's interpreter range, pinned package version, and core
import without loading credentials or starting research.

When the app opens, use Research setup to select or create a Provider Profile.
Each profile independently selects its model, search, and embedding providers.
The default profile maps DeepSeek to the GPT Researcher model layer, Tavily to
retrieval, and Ollama to local document embeddings. OpenAI,
OpenAI-compatible endpoints, DuckDuckGo, and fully local Ollama configurations
are also supported.

Non-secret profile metadata is persisted as
`research-provider-profiles.json` under the app-data directory. Wutai stores
keys in the system keychain through `keyring-rs`, scoped by profile, provider,
and purpose, then injects only the active profile into the sidecar process. The
developer environment variables `DEEPSEEK_API_KEY`, `OPENAI_API_KEY`, and
`TAVILY_API_KEY` remain fallbacks for scripted runs. Provider metadata, never
secret values, is attached to each successful run's audit payload.

The adapter script path can be overridden with
`WUTAI_GPT_RESEARCHER_ADAPTER_SCRIPT`.

## Evidence Gate v0.1

Evidence Gate runs after GPT Researcher drafts the report. It reuses the active
Provider Profile model to extract at most 30 decision-relevant claims into a
strict JSON shape. Model output is treated as untrusted input: Wutai drops
unknown fields and URLs, derives claim risk from fixed categories, classifies
source provenance with deterministic hostname rules, and computes the final
gate status itself.

High-risk claims include licenses, pricing, release dates, adoption metrics,
and product identity. These require both model-assessed support and a captured
primary or repository source to pass. Locked regressions currently reject the
known incorrect claims that OpenClaw is Apache 2.0 and that Multica uses
unmodified Apache 2.0. They are regression seeds, not a universal fact
database.

The gate returns `pass`, `warning`, or `fail`. A non-pass result does not discard
the report; the task becomes `completed_with_warnings`, the UI displays the
Evidence Gate summary, and `claims.json` plus `verification.json` preserve the
review surface. A pass means the extracted claims met the configured evidence
rules. It is not a guarantee that every statement in the report is true.

## Setup Preflight

When the GPT Researcher adapter is selected, Wutai calls the Tauri
`check_gpt_researcher` command during startup. The command returns:

- `ready`
- `summary`
- per-check setup status
- plain-language fixes
- detected Python path, script path, and GPT Researcher package version when
  available

The preflight currently checks:

- local sidecar script availability
- Python launchability
- `gpt-researcher` package availability for that Python
- active Provider Profile validity
- required model access from the profile keychain slot or matching environment
  variable
- required search access from the profile keychain slot or matching environment
  variable
- required embedding access, including model-key reuse for OpenAI profiles
- Ollama endpoint reachability when the active profile uses Ollama

If any required check fails, Wutai blocks creation of new real research tasks
and shows the setup summary in the task console. The mock adapter returns a
ready preflight with no visible setup panel, so the default offline workflow
stays unchanged.

The credential, preflight, and cancellation commands have Rust tests through
Tauri's official mock runtime and WebView invoke handler. Credential tests use
an in-memory implementation of the same storage interface and do not read or
write the user's system keychain. The process test starts a local sleeping
child process, cancels it by task ID through IPC, and verifies termination and
registry cleanup. The suite does not call external model or search APIs and
does not run a real GPT Researcher task.

The streaming reader test uses an in-memory Tauri Channel. It verifies ordered
stage and log delivery, malformed stage fallback, provider-key redaction, and
preservation of captured audit lines without launching Python or external
services.

The Provider Profiles Playwright contract test supplies a mocked Tauri command
boundary, verifies the default and Advanced controls, switches model provider,
and checks the configured 860-pixel desktop minimum width for horizontal
overflow.

`tests/python/test_evidence_gate.py` runs without network or model access. It
locks source classification, fenced JSON parsing, primary-evidence warnings,
and OpenClaw/Multica license regressions.

## Storage Boundary

The scaffold uses SQLite when running in Tauri and `localStorage` when running
as a plain web app.

The UI depends on the `TaskStore` interface rather than the storage backend.
This keeps the browser preview testable while allowing the desktop shell to use
real local persistence.

## Artifact Boundary

The mock adapter still generates artifact content in memory. When running in
Tauri, `artifactWriter` calls the `write_task_artifacts` command and writes
files to:

```text
<app-data-dir>/artifacts/<task_id>/
```

When running as a plain web app, artifacts remain available in preview and
download mode only.

3. Keep the mock adapter available for offline UI/e2e validation.
